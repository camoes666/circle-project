  
슈퍼파워즈 스킬 가이드

# 슈퍼파워즈 스킬 가이드

## 소개

슈퍼파워즈는 Claude가 일을 더 체계적이고 효율적으로 하도록 돕는 14개의 실용적인 기법입니다. 이 가이드는 프로그래밍 경험이 없는 분들을 위해 각 스킬을 쉽고 친절하게 설명합니다.

생각해보면 스킬들은 네 가지 카테고리로 나뉩니다:

1. **기획·설계** — 일을 시작하기 전에 계획을 세우는 과정
2. **개발 워크플로우** — 실제 일을 진행할 때 따르는 순서와 방법들
3. **검증·디버깅** — 일이 제대로 되었는지 확인하고 문제를 찾아 고치는 방법
4. **협업·코드리뷰** — 다른 사람들과 일하고 피드백을 주고받는 방식
5. **메타** — 스킬 자체를 이해하고 만드는 방법

---

## 스킬 목록 (기획·설계)

### 1. Brainstorming (브레인스토밍)

**언제 쓰나**  
새로운 기능을 만들거나, 문제를 해결해야 할 때 실제 작업을 시작하기 전에 사용합니다.

**뭘 하나**  
Claude와 함께 대화하면서 아이디어를 구체적인 계획으로 만듭니다. 질문을 통해 필요한 것이 뭔지 명확히 하고, 가능한 방법들을 비교한 다음, 최종 설계를 문서화합니다. 사용자의 승인을 받은 후에만 실제 작업을 시작합니다.

**특징·장점**  
단순해 보이는 일도 설계 과정을 거치면 훨씬 나은 결과가 나옵니다. 설계 문서는 나중에 참고할 수 있는 자산이 됩니다.

**수강생 활용 팁**  
"좀 간단해 보이는 일인데 설계가 필요한가?"라고 생각될 때야말로 설계를 꼭 해야 할 때입니다.

---

### 2. Writing Plans (계획 작성)

**언제 쓰나**  
브레인스토밍으로 설계가 승인된 후, 실제 작업 단계들을 정리할 때 사용합니다.

**뭘 하나**  
설계를 토대로 상세한 구현 계획을 만듭니다. 어떤 파일들을 수정할지, 각 단계마다 정확히 무엇을 해야 할지를 단계별로 기록합니다. 계획의 각 단계는 2-5분 정도로 끝낼 수 있는 크기여야 합니다.

**특징·장점**  
자세한 계획이 있으면 실수를 줄이고, 진행 상황을 명확하게 추적할 수 있습니다. 계획을 따르면 일이 빠르고 확실하게 진행됩니다.

**수강생 활용 팁**  
"계획? 그냥 하면 되지 않나?"라고 느껴질 수 있지만, 계획이 있으면 더 빠르고 정확합니다.

---

## 스킬 목록 (개발 워크플로우)

### 3. Using Git Worktrees (깃 작업 공간 만들기)

**언제 쓰나**  
새로운 기능 작업을 시작하거나 계획을 실행하기 전에, 현재 작업과 분리된 독립적인 작업 공간이 필요할 때 사용합니다.

**뭘 하나**  
"깃 작업 공간"이라는 것을 만들어서, 여러 작업을 동시에 진행할 수 있게 합니다. 각 작업 공간은 독립적이어서, 한 작업이 다른 작업에 영향을 주지 않습니다. 프로젝트에 맞게 필요한 도구들을 자동으로 설치하고, 테스트를 실행해서 처음부터 제대로 시작되는지 확인합니다.

**특징·장점**  
작업 공간이 분리되어 있어서 실수할 염려가 줄어듭니다. 여러 기능을 동시에 작업할 때 특히 유용합니다.

**수강생 활용 팁**  
"공간을 나눈다"는 개념이 낯설 수 있지만, 실생활의 책상 여러 개를 준비하는 것과 비슷합니다.

---

### 4. Test-Driven Development (테스트 먼저 만들기, TDD)

**언제 쓰나**  
새로운 기능을 만들거나 버그를 고칠 때, 코드를 쓰기 전에 사용합니다.

**뭘 하나**  
먼저 "테스트"(이게 잘 작동하나 확인하는 자동화된 검사)를 만들어서 실패하는 것을 본 후, 그 테스트를 통과시킬 정도의 최소한의 코드만 씁니다. 그 다음에 코드를 정리합니다. 이 주기를 반복합니다.

**특징·장점**  
코드를 쓰기 전에 테스트를 만들면, 만든 코드가 정말 필요한 일을 하는지 확실합니다. 나중에 다른 코드를 바꿀 때도 테스트가 있으면 안심입니다.

**수강생 활용 팁**  
"테스트를 먼저?"라고 놀랄 수 있지만, 실제로는 이렇게 하는 게 더 빠르고 실수를 줄입니다.

---

### 5. Executing Plans (계획 실행하기)

**언제 쓰나**  
작성된 구현 계획을 단계별로 따라서 실행할 때 사용합니다.

**뭘 하나**  
계획 파일을 읽고 검토한 후, 계획의 각 작업을 하나씩 진행합니다. 각 단계마다 진행 상황을 표시하고, 계획에 나온 검증을 실행합니다. 모든 작업이 끝나면 마무리 과정을 진행합니다.

**특징·장점**  
계획을 체계적으로 따르기 때문에 빠뜨리는 부분이 없습니다. 진행 상황을 명확하게 추적할 수 있습니다.

**수강생 활용 팁**  
계획이 명확하면 실행은 어렵지 않습니다. 각 단계를 정확히 따르면 됩니다.

---

### 6. Subagent-Driven Development (전담 도우미와 함께 개발하기)

**언제 쓰나**  
구현 계획이 있고, 각 작업이 서로 독립적일 때 사용합니다. 특히 현재 대화를 계속하면서 작업을 진행할 때 좋습니다.

**뭘 하나**  
계획의 각 작업을 위해 전담 Claude(도우미)를 배정합니다. 각 도우미는 작업을 끝낸 후, 다른 도우미가 그 결과를 검수합니다. 검수는 두 단계로 나뉩니다: 먼저 설계대로 잘 만들어졌는지, 다음으로 코드의 품질이 좋은지 확인합니다.

**특징·장점**  
전담 도우미를 사용하면 작업이 더 빠르고 정확합니다. 검수 단계가 두 번 있어서 문제를 빨리 찾아 고칠 수 있습니다. 현재 대화를 계속할 수 있다는 장점도 있습니다.

**수강생 활용 팁**  
"도우미를 여러 명 배정한다"고 생각하면 됩니다. 각 도우미는 자신의 작업에만 집중합니다.

---

### 7. Dispatching Parallel Agents (여러 도우미 동시에 일시키기)

**언제 쓰나**  
독립적인 문제들이 여러 개 있어서, 서로 다른 도우미가 동시에 다양한 문제를 조사할 때 사용합니다.

**뭘 하나**  
서로 다른 문제들을 여러 Claude 도우미에게 동시에 넘깁니다. 각 도우미는 자신의 문제만 집중해서 파악합니다. 문제들이 서로 독립적이면, 모두가 동시에 일하니까 훨씬 빠릅니다.

**특징·장점**  
여러 도우미가 동시에 일하니까 시간이 훨씬 줄어듭니다. 각 도우미는 자신의 문제에만 집중하니까 실수할 가능성도 줄어듭니다.

**수강생 활용 팁**  
"세 가지 일을 셋에게 나눠주면, 한 사람이 셋을 하는 것보다 훨씬 빠르다"는 생각입니다.

---

## 스킬 목록 (검증·디버깅)

### 8. Systematic Debugging (체계적으로 문제 찾기)

**언제 쓰나**  
버그를 발견했거나 테스트가 실패했을 때, 문제의 원인을 찾기 위해 사용합니다.

**뭘 하나**  
먼저 에러 메시지를 꼼꼼히 읽고, 문제를 재현해봅니다. 최근에 뭐가 바뀌었는지 확인합니다. 문제가 어디서 발생하는지 데이터를 모아 추적합니다. 그 다음 원인이 뭔지 가설을 세워 테스트합니다. 원인을 찾으면 그것을 고칩니다.

**특징·장점**  
원인을 찾고 고치면 같은 문제가 다시 생기지 않습니다. 임시방편으로 고치는 것보다 훨씬 더 나은 해결책입니다.

**수강생 활용 팁**  
"무조건 원인부터 찾자"는 마음가짐이 중요합니다. "일단 고쳐보고 봤자"는 위험합니다.

---

### 9. Verification Before Completion (완성하기 전에 확인하기)

**언제 쓰나**  
일이 완성되었다고 주장하기 전에, 정말 그런지 확인할 때 사용합니다.

**뭘 하나**  
"완성되었다"고 말하기 전에, 그것을 확인하는 명령어를 실행하고 결과를 읽습니다. 테스트가 통과하는지, 빌드가 성공하는지 같은 것들을 직접 확인하고, 그 증거를 보여줍니다.

**특징·장점**  
증거를 보이니까 신뢰할 수 있습니다. 실수를 사전에 방지합니다. 검증을 하고 나서 일을 제출하니까 품질이 높습니다.

**수강생 활용 팁**  
"확신이 들어"가 아니라 "실제로 확인했어"가 중요합니다.

---

## 스킬 목록 (협업·코드리뷰)

### 10. Requesting Code Review (코드 검수 요청하기)

**언제 쓰나**  
작업을 끝낸 후, 다른 Claude가 코드를 검토해주길 원할 때 사용합니다.

**뭘 하나**  
작업 내용, 뭘 구현했는지, 어떤 요구사항을 충족했는지 등을 정리해서 전담 검수 Claude에게 넘깁니다. 검수자는 코드가 좋은지, 요구사항을 충족하는지 확인합니다. 검수자가 지적한 문제들을 고칩니다.

**특징·장점**  
다른 관점으로 코드를 보니까 놓친 부분을 찾을 수 있습니다. 검수는 자주 할수록 좋습니다.

**수강생 활용 팁**  
"자기 일만 했다고 생각하면 안 된다. 남이 한번 봐줘야 한다"는 마음으로 요청하면 됩니다.

---

### 11. Receiving Code Review (코드 검수 받기)

**언제 쓰나**  
검수자가 제시한 피드백을 받았을 때, 어떻게 대응할지 생각할 때 사용합니다.

**뭘 하나**  
피드백을 읽고, 정확히 뭘 말하는 건지 이해합니다. 검수자가 맞는지 코드를 확인해봅니다. 진짜 필요한 것이면 고칩니다. 이상하다면 기술적인 이유를 들어 질문합니다. 처음부터 고쳐야 할 모든 것들을 파악한 후, 하나씩 고칩니다.

**특징·장점**  
피드백을 무작정 따르지 않고 검증합니다. 검수자도 실수할 수 있으니까요. 기술적으로 맞는지를 먼저 확인합니다.

**수강생 활용 팁**  
"고마워요!"라고 말하지 말고, 조용히 고칩니다. 고쳐진 코드가 답입니다.

---

## 스킬 목록 (마무리)

### 12. Finishing a Development Branch (개발 끝내기)

**언제 쓰나**  
모든 작업이 끝나고 테스트가 통과한 후, 결과를 어떻게 정리할지 결정할 때 사용합니다.

**뭘 하나**  
먼저 테스트가 정말 통과하는지 확인합니다. 그 다음 네 가지 선택지를 보여줍니다:

1. 현재 작업을 메인 브랜치에 합치기
2. 온라인에 올려서 다른 사람의 검토 요청하기
3. 지금은 안 하고 나중에 처리하기
4. 이 작업 버리기

사용자가 선택하면 그에 맞게 정리합니다.

**특징·장점**  
어떤 방식으로든 작업을 깔끔하게 정리할 수 있습니다. 선택지가 명확해서 혼동이 없습니다.

**수강생 활용 팁**  
"이제 뭐할 거야?"라는 질문이 나올 때 쓰면 됩니다.

---

## 스킬 목록 (메타)

### 13. Using Superpowers (슈퍼파워즈 사용하기)

**언제 쓰나**  
일을 시작하기 전에, 사용할 수 있는 스킬이 있는지 확인할 때 사용합니다.

**뭘 하나**  
스킬 목록을 검색해서 현재 작업에 맞는 스킬이 있는지 찾습니다. 맞는 스킬이 있으면 그것을 불러옵니다. 스킬의 내용을 정확하게 따릅니다.

**특징·장점**  
올바른 스킬을 찾으면 작업이 훨씬 체계적이 됩니다. 스킬들이 검증된 기법이니까 믿을 수 있습니다.

**수강생 활용 팁**  
"이 일에 맞는 기법이 있을까?"라고 자주 생각하면, 더 좋은 결과를 얻습니다.

---

### 14. Writing Skills (스킬 만들기)

**언제 쓰나**  
새로운 기법을 만들거나, 기존 스킬을 수정할 때 사용합니다. 전문가만 사용할 스킬입니다.

**뭘 하나**  
테스트 먼저 만들기(TDD)를 문서에 적용합니다. 먼저 스킬 없이 시도해봐서 실패합니다. 그 다음 스킬을 만들어서 성공하게 합니다. 마지막으로 허점을 막아서 더 좋게 만듭니다.

**특징·장점**  
스킬을 만들 때도 체계적으로 하니까, 정말 도움이 되는 스킬이 만들어집니다.

**수강생 활용 팁**  
40명 중 대부분은 스킬을 만들 필요가 없습니다. 기존 스킬을 잘 사용하는 것이 더 중요합니다.

---

## 추천 사용 순서

처음 슈퍼파워즈를 배우시는 분들께 추천하는 순서입니다:

1. **Using Superpowers** — 먼저 스킬이 뭔지, 어떻게 찾는지 이해합니다.
2. **Brainstorming** — 새로운 일을 시작할 때마다 기획부터 시작하는 습관을 들입니다.
3. **Writing Plans** — 설계가 나온 후, 구체적인 단계들을 정리하는 법을 배웁니다.
4. **Test-Driven Development** — 코드를 쓸 때 테스트를 먼저 만드는 방식을 익힙니다.
5. **Verification Before Completion** — 일이 끝났다고 주장하기 전에, 항상 확인하는 습관을 들입니다.
6. **Systematic Debugging** — 문제가 생겼을 때 원인부터 찾는 방법을 배웁니다.
7. **Using Git Worktrees** — 작업 공간을 분리하는 방법을 배웁니다.
8. **Executing Plans** — 계획을 따라 일을 진행하는 방법을 배웁니다.

나머지 스킬들은 필요할 때마다 배우면 됩니다:

- **코드리뷰가 필요할 때:** Requesting Code Review, Receiving Code Review
- **여러 사람과 함께 일할 때:** Subagent-Driven Development, Dispatching Parallel Agents
