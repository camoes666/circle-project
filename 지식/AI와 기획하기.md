## 오늘 실습에서 만들 것

세 개의 스킬을 연달아 써서 **기획서 → 스펙 → 플랜**까지 문서 세트를 완성합니다.

1. **design-thinking 스킬** — 제품 기획서 + Lean Canvas (+ 조건부 추가 문서)를 만든다 (강사 제작)
2. **brainstorming 스킬** — 개발 스펙(Design Doc)을 만든다 (superpowers 패키지에 포함)
3. **writing-plans 스킬** — 구현 플랜을 만든다 (superpowers 패키지에 포함, brainstorming이 자동 호출)

끝나면 여러분 프로젝트 폴더 안에 이런 문서들이 생겨요. (아래는 **최대치** 기준)

```
~/Documents/programming/my-project/
└─ docs/
   ├─ product/                                    ← design-thinking 산출물 (2~5개)
   │  ├─ 2026-04-15-커리어코치-기획서.md           [필수]    "왜·누구·무엇"
   │  ├─ 2026-04-15-커리어코치-lean-canvas.md      [필수]    비즈니스 모델 9칸
   │  ├─ 2026-04-15-커리어코치-경쟁분석.md         [조건부]  차별화 근거
   │  ├─ 2026-04-15-커리어코치-사용자플로우.md     [조건부]  핵심 인터랙션 흐름
   │  └─ 2026-04-15-커리어코치-로드맵.md           [조건부]  MVP 이후 발전 방향
   └─ superpowers/
      ├─ specs/                                   ← brainstorming 산출 (설계도)
      │  └─ 2026-04-15-mvp-design.md             [필수]    "어떻게 설계되어 있나"
      └─ plans/                                   ← writing-plans 산출 (공사 일정표)
         └─ 2026-04-15-mvp-plan.md               [필수]    "오늘 뭘 할까"
```

_▲ 예시: 실습 완료 후 생성되는 최대 7개 문서의 폴더 구조_  
![](https://share.note.sx/files/yz/yz921ctcntz11glfxu4w.png)

총 **최소 4개 ~ 최대 7개** 문서가 만들어져요.

> 💡 **왜 어떤 때는 4개, 어떤 때는 7개?**  
> 기획서·Lean Canvas·Design Doc·Plan 이 4개는 **항상** 만들어져요. 나머지 3개(경쟁 분석 / 사용자 플로우 / 로드맵)는 Claude가 대화 내용을 보고 **제품 복잡도에 따라** 필요하다고 판단할 때 추가로 제안합니다.

|추가 문서|추가되는 조건|
|---|---|
|경쟁 분석|경쟁 서비스가 2개 이상 등장하고 차별화 근거가 대화에서 나옴|
|사용자 플로우|핵심 인터랙션이 복잡하거나 플로우가 2단계 이상|
|로드맵|MVP 이후 발전 방향이 대화에서 언급됨|

세 조건에 다 걸리면 7개, 하나도 안 걸리면 4개가 나옵니다. 대부분의 실제 서비스 아이디어는 **5~6개** 사이로 나와요.

---

## 🎯 `product` 문서들은 "개발용"이 아니라 "서비스용"입니다

여기에 만들어진 기획서 · Lean Canvas · 경쟁 분석 · 사용자 플로우 · 로드맵은 **개발을 위한 문서가 아니에요.** 제품 전체의 뼈대를 잡는 문서입니다.

이 수업의 목표는 **"코드를 짜는 것"이 아니라 "서비스를 만들어 사업화하는 것"** 이에요. 그래서 오늘 만든 문서들은 앞으로 이렇게 쓰입니다.

|다음 단계|이 문서를 어떻게 쓰나|
|---|---|
|**개발**|기획서 → brainstorming 스펙 → 실제 코드|
|**랜딩페이지 카피**|기획서의 WHO/WHAT → "누구를 위한 / 어떤 문제를 푸는지" 후킹 문구로 변환|
|**홈페이지 · 소개 페이지**|Lean Canvas의 고유 가치 제안 → 메인 히어로 섹션 카피|
|**블로그 · SNS 콘텐츠**|페르소나의 페인포인트 → 공감 글감, JTBD → 사용 사례 스토리|
|**유저 인터뷰 · 설문**|페르소나 + Problem Statement → 검증할 가설 도출|
|**투자 자료 · IR**|Lean Canvas 9칸 → 피치덱의 뼈대 그대로|
|**광고 · 마케팅 소재**|사용자 플로우 → "처음 써보는 사람이 느낄 impact 한 장면" 추출|
|**로드맵 공유**|로드맵 문서 → 초기 유저 대상 "다음엔 이게 나와요" 공지|

> 💡 **핵심**: 오늘 대화로 뽑아낸 한 줄 한 줄이, 나중에 **판매 페이지 카피**, **투자자에게 할 설명**, **유저 인터뷰 질문**이 돼요. 그래서 대충 넘기지 말고, **나중의 내가 꺼내 쓸 수 있을 만큼 구체적으로** 답해주세요.

> ⚠️ 기획서의 품질 = 앞으로 만들 모든 서비스 자산의 품질입니다. 개발만 생각하고 적당히 답하면, 5회차에서 랜딩페이지 쓸 때 "어… 우리 서비스가 정확히 뭐더라…" 에서 또 막혀요.

# STEP 1. 프로젝트 폴더 만들고 Claude Code 실행하기

## 1-1. 프로젝트 폴더 만들기

### 왜 먼저 `programming` 폴더를 만드나?

바로 `Documents` 안에 프로젝트 폴더를 만드는 것도 가능하지만, `Documents`에는 이미 워드 파일, 사진, 다운로드받은 자료 등 **온갖 잡다한 폴더**가 섞여 있어서 나중에 프로젝트를 찾기 어려워져요.

그래서 이렇게 **2단계로** 만드는 걸 강하게 권장합니다.

```
Documents/
└─ programming/          ← 개발 프로젝트만 모아두는 폴더 (1번만 만들면 됨)
   ├─ my-project/        ← 이번 수업 프로젝트
   ├─ (다음 프로젝트)
   └─ (그 다음 프로젝트)
```

이렇게 해두면 앞으로 만드는 **모든 개발 프로젝트가 한곳에** 모여서, Claude Code를 열 때도 `~/Documents/programming` 안에서 고르면 됩니다. Finder나 파일 탐색기에서도 바로 찾을 수 있고요.

### 터미널 클로드 코드 사용시

#### 폴더 만들기

**Mac**: 터미널 열고 아래 처럼 입력하면 개발자처럼 보입니다!

```bash
# programming 폴더가 없으면 먼저 만들기 (한 번만)
mkdir -p ~/Documents/programming

# 이번 프로젝트 폴더 만들고 들어가기
cd ~/Documents/programming
mkdir my-project
cd my-project
```

![](https://share.note.sx/files/p1/p1p7xnaxa4yr9tvn9tlh.jpg)  
![](https://share.note.sx/files/6q/6q8joy4zpgrvgshog29g.jpg)

**Windows**: PowerShell 열고 아래 입력

```powershell
# programming 폴더가 없으면 먼저 만들기 (한 번만)
New-Item -ItemType Directory -Force -Path $HOME\Documents\programming

# 이번 프로젝트 폴더 만들고 들어가기
cd $HOME\Documents\programming
mkdir my-project
cd my-project
```

> 윈도우는 스크린샷이 없네요. 파일탐색기에서 원하는 경로 지정해서 사용해주세요! 터미널 열었을때 기본 폴더위치에서 너무 먼 경로에 작업폴더를 만들면 약간 부편하실 수 있습니다!

> 💡 `mkdir -p` / `-Force` 옵션은 "이미 폴더가 있으면 그냥 넘어가라"는 뜻이에요. 다음에 두 번째 프로젝트를 만들 땐 이 한 줄을 지워도 되고, 그대로 둬도 에러 안 납니다.

💡 **왜 이 폴더가 중요한가요?**

Claude Code는 "현재 폴더"를 작업 공간으로 인식해요. 기획서, 스펙, 코드가 전부 이 폴더 안에 쌓입니다. 즉 **`my-project/` = 여러분의 현재 프로젝트 전체**예요. 그리고 **`programming/` = 앞으로 만들 모든 프로젝트의 집**이고요.

# STEP 3. design-thinking 스킬 설치하기

이 스킬은 강사가 직접 만든 커스텀 스킬이라 GitHub에 없어요. **파일을 직접 복사**해서 스킬 폴더에 넣어야 합니다. 먼저 **로컬 설치**로 사용해보시고 앞으로도 계속 쓰시겠다 싶을 때 **글로벌 설치**로 올리는 걸 권장합니다.

## 3-1. design-thinking.md 파일 내용 복사해두기

## 3-2. Claude에게 스킬로 만들어달라고 요청하기


> 💡 **로컬 vs 글로벌**

> - **글로벌**: 모든 프로젝트에서 사용 가능. 계속 쓸 것 같다 싶으면 `글로벌 스킬로 만들어줘`로 변경.

# STEP 4. design-thinking 스킬로 기획서 만들기

이제 진짜 실습. 여러분의 아이디어를 기획서로 바꿉니다.

## 4-1. 스킬 호출하기

Claude Code에 아래처럼 입력.

![](https://share.note.sx/files/cg/cgkvxckyplwjkrwjoi9r.jpg)  
디자인싱킹 스킬을 호출해서 챗봇에게 어떤 제품, 어떤 서비스를 만들지 의논해보세요. 디테일할수록 더 정교한 기획서가 작성됩니다.

## 4-2. 대화에 답하기

Claude가 한 번에 하나씩 질문을 던져요. 막히면 **"네가 추천해줘"** 라고 말해도 OK입니다.

질문은 대략 이 순서로 나와요.

1. **문제 발견** — 누가 어떤 불편을 겪나요? 기존 해결 시도는? 왜 지금 이 문제를?
2. **문제 정의** — 페르소나(가상 인물) 확정 → JTBD → Problem Statement
3. **아이데이션** — Claude가 2~3가지 해결 방향을 제안 → 여러분이 고름
4. **우선순위화** — MoSCoW(Must/Should/Could/Won't)로 MVP 확정

> 💡 **답이 안 떠오를 때**
> 
> - "2~3개 선택지로 줘" → 열린 질문을 객관식으로 바꿔달라
> - "네가 추천해줘" → Claude의 의견을 먼저 들어보기
> - "이 질문 왜 물어봐?" → 질문의 의도를 설명해달라
> 
> 자연스러운 대화라고 생각하세요. 면접 보는 게 아니에요.

## 4-3. 기획서 작성 단계 진입

네 단계 대화가 끝나면 Claude가 **"이제 기획서와 Lean Canvas를 파일로 작성하겠습니다"** 라고 알려줘요.

그리고 `docs/product/` 폴더 안에 **기본 2개 파일**을 만듭니다.

- `YYYY-MM-DD-<제품명>-기획서.md`
- `YYYY-MM-DD-<제품명>-lean-canvas.md`

![](https://share.note.sx/files/hb/hbu5206tvhox2z312kbo.png)

### 추가 문서 제안 (조건부)

이어서 Claude가 여러분 아이디어의 복잡도를 판단해서, 필요하다고 보면 **추가 문서를 제안**하거나 생략해도 된다고 말하기도 합니다.

> "기획서와 Lean Canvas 외에 **경쟁 분석 / 사용자 플로우 / 로드맵**도 작성하면 개발 스펙 작성에 도움이 됩니다. 작성할까요?"

![](https://share.note.sx/files/hc/hc6bhpe2sx92eo4leaaa.png)

**"다 만들어줘"** 라고 답하면 나머지 문서까지 `docs/product/` 에 이어서 생성됩니다.

|추가 문서|들어가는 내용|
|---|---|
|경쟁 분석|대안 서비스 목록, 각자의 한계, 우리 제품의 차별점|
|사용자 플로우|온보딩 → 핵심 기능 사용 → 이탈까지의 단계별 흐름|
|로드맵|MVP 이후 언제 뭘 추가할지 타임라인|

> 💡 **헷갈릴 때는**: 제안받은 문서 중 **뭘 만들지 고민되면 "네가 추천해줘"** 라고 답하세요. Claude가 대화 맥락을 보고 필요한 것만 골라 만듭니다.

> ⚠️ **추가 문서 제안이 안 나오는 경우도 정상입니다.** 아이디어가 단순하면 기본 2개만 나오고 바로 검토 단계로 넘어가요. 이건 스킬이 "더 안 만들어도 충분하다"고 판단했다는 뜻이에요.

## 🔧 스펙과 플랜은 각각 어떤 역할인가

기획서가 "무엇을 왜 만드는가"였다면, **스펙과 플랜은 "어떻게 만드는가"의 두 층**이에요. 헷갈리는 분들이 많아서 정리해둘게요.

### Design Doc (스펙) — "설계도"

`docs/superpowers/specs/...-design.md`

**무엇이 들어있나**

- 어떤 기술 스택으로 만들지 (예: Next.js + localStorage)
- 데이터 모델 (어떤 정보를 어떻게 저장하는지)
- 주요 컴포넌트·함수 구조
- 외부 API / 라이브러리 의존성
- 에러 처리, 보안, 성능 고려사항

**언제 어떻게 쓰나**

|쓰임새|어떻게|
|---|---|
|**AI와 코딩할 때**|"이 스펙 그대로 구현해줘" — AI에게 일관된 구현 맥락 제공|
|**중간에 방향을 잃었을 때**|"원래 스펙이 뭐였지?" 로 돌아와 기준점 확인|
|**다른 사람(또는 미래의 나)에게 설명할 때**|"왜 이 구조로 만들었는지"의 근거 문서|
|**기술 스택 바꿀 때**|같은 스펙을 다른 스택으로 다시 구현하기 위한 명세|
|**디버깅할 때**|"원래 이 함수가 뭘 해야 하는지"의 기준|
|**리팩토링할 때**|"지금 코드가 스펙에서 벗어났나?" 비교 기준|

> 💡 **스펙은 "완성된 이후"에도 살아있는 문서예요.** 서비스를 운영하면서 기능이 바뀔 때마다 스펙을 먼저 업데이트하고, 그걸 기반으로 코드를 고치는 게 정석이에요.

### Plan (플랜) — "공사 일정표"

`docs/superpowers/plans/...-plan.md`

**무엇이 들어있나**

- 구현 순서 (1단계 → 2단계 → 3단계…)
- 각 단계에서 만들/수정할 파일 목록
- 단계별 완료 기준 ("이게 되면 이 단계 끝")
- 의존성 (뭘 먼저 해야 뭘 할 수 있는지)

**언제 어떻게 쓰나**

|쓰임새|어떻게|
|---|---|
|**AI와 코딩 시작할 때**|"플랜의 1단계부터 시작해줘" — AI가 큰 작업을 안전한 단위로 분할|
|**어디까지 했는지 추적**|완료한 단계에 체크, 오늘 할 범위를 명확히|
|**AI가 헷갈릴 때**|"지금 플랜 몇 단계야?" 로 현 위치 복기|
|**중단 후 재개할 때**|다음 세션에서 "플랜 3단계부터 이어가줘" 로 바로 복귀|
|**범위 통제**|플랜에 없는 작업이 튀어나오면 "이건 플랜에 없어, 나중에" 로 거절|
|**진척도 공유**|팀원·투자자에게 "지금 플랜의 2/5 단계" 식으로 설명|

> 💡 **플랜은 "소모품"에 가까워요.** 프로젝트 진행 중에 다 쓰고 나면 새 기능을 추가할 때 또 새 플랜을 만듭니다. 스펙은 장기 자산, 플랜은 단기 작업지시서라고 생각하면 쉬워요.

### 둘의 관계 한눈에

```
기획서·Lean Canvas  ─→  Design Doc(스펙)  ─→  Plan(플랜)  ─→  실제 코드
    "왜·누구·무엇"        "어떻게 설계되어 있나"     "오늘 뭘 할까"      "동작하는 제품"
    서비스의 헌법           제품의 설계도            공사 일정표
```

> ⚠️ **이 순서를 건너뛰지 마세요.** 기획서 없이 스펙 쓰면 "누구를 위한 기능인지 모르는 기능"이 생기고, 스펙 없이 플랜 쓰면 "일정은 있는데 뭘 만드는지 모호한" 상태가 됩니다. 스펙 없이 AI한테 바로 코딩 시키는 게 1회차에서 2% 부족했던 이유예요.

# STEP 6. 최종 산출물 점검

여기까지 끝났으면 프로젝트 폴더가 이렇게 돼 있어야 합니다.

```
~/Documents/programming/my-project/
└─ docs/
   ├─ product/                                    ← design-thinking 산출물 (2~5개)
   │  ├─ YYYY-MM-DD-<제품명>-기획서.md            [필수]    "왜·누구·무엇"
   │  ├─ YYYY-MM-DD-<제품명>-lean-canvas.md       [필수]    비즈니스 모델 9칸
   │  ├─ YYYY-MM-DD-<제품명>-경쟁분석.md          [조건부]  차별화 근거
   │  ├─ YYYY-MM-DD-<제품명>-사용자플로우.md      [조건부]  핵심 인터랙션 흐름
   │  └─ YYYY-MM-DD-<제품명>-로드맵.md            [조건부]  MVP 이후 발전 방향
   └─ superpowers/
      ├─ specs/                                   ← brainstorming 산출 (설계도)
      │  └─ YYYY-MM-DD-<주제>-design.md          [필수]    "어떻게 설계되어 있나"
      └─ plans/                                   ← writing-plans 산출 (공사 일정표)
         └─ YYYY-MM-DD-<주제>-plan.md            [필수]    "오늘 뭘 할까"
```

총 문서 수는 **최소 4개 ~ 최대 7개**까지 나올 수 있어요.

Claude Code에서 한번에 확인하는 법:

```
docs 폴더 전체 구조를 트리로 보여줘
```

# FAQ

## Q1. 스킬이 자동으로 호출되지 않아요

**증상**: "design-thinking 스킬을 써줘"라고 했는데 Claude가 그냥 일반 답변을 해요.

**해결**:

1. `/exit`으로 Claude Code를 끄고 다시 `claude` 실행 (스킬 파일은 실행 시점에 읽혀요)
2. 스킬 파일 경로가 정확히 `~/.claude/skills/design-thinking/SKILL.md` 인지 확인
3. 파일 첫 줄이 `---` 로 시작하고 `name: design-thinking` 이 포함됐는지 확인

## Q2. 질문이 너무 많아서 지쳐요

**해결**: 중간에 이렇게 말하세요.

```
질문이 많네. 지금까지 답변한 걸 기반으로
네가 판단해서 나머지는 채워줘. 초안을 먼저 보고 싶어.
```

Claude가 합리적 추론으로 빈칸을 채운 초안을 먼저 보여줍니다. 틀린 부분만 고치면 돼요.

## Q3. 아이디어가 아직 모호해요

**해결**: 모호한 채로 그냥 시작하세요. design-thinking의 1단계가 **문제 발견**이에요. Claude의 질문에 답하는 과정에서 아이디어가 구체화됩니다. "아직 잘 모르겠어" 도 답변으로 인정돼요.

## Q4. 파일이 어디 생기는지 모르겠어요

**해결**: Claude에게 물어보세요.

```
지금까지 이 세션에서 만든 파일들의 절대 경로를 알려줘
```

Claude가 `/Users/.../Documents/programming/my-project/docs/product/...` 형태로 알려줍니다.

## Q5. 결과 문서가 마음에 안 들어요

**해결**: **승인하지 마세요.** 구체적으로 지적하면 Claude가 그 부분만 고칩니다.

나쁜 지적: "뭔가 어색해"  
좋은 지적: "페르소나의 '3년차 마케터' 대신 '프리랜서 디자이너'로 바꿔줘. 그리고 JTBD 문장이 길어. 한 문장으로 줄여줘."
